### 1. Introduction — The Professional Software Engineer in the AI Era

#### 1.1 Who is a Professional Software Engineer?
A professional software engineer is not just someone who knows how to write code, but someone who understands the full software development life cycle, knows how to design, document, test, and maintain complex systems. A professional engineer adheres to high code quality standards, follows proven best practices, and is capable of working in teams while maintaining consistency, clear communication, and accountability.

**Core characteristics of a professional software engineer include:**
*   **Systems Thinking:** The ability to see the "big picture" and understand how each component integrates into the whole system.
*   **Planning Before Execution:** Writing requirements documents, architecture, and design before writing the first line of code.
*   **Zero-Compromise Quality:** Commitment to writing clean, documented code, covered by comprehensive and secure tests.
*   **Continuous Learning:** Staying updated with technologies, tools, and new methodologies.
*   **Effective Communication:** The ability to describe technical solutions to diverse audiences.

#### 1.2 Software Development Teamwork
Professional software development is almost always performed in teams. Effective teamwork is based on several key principles:
*   **Roles and Responsibilities:** A typical development team includes a Software Architect (responsible for system design), Developers (who implement the code), a QA Engineer (ensuring product quality), a Product Manager (defining requirements), and sometimes a DevOps engineer (responsible for infrastructure and deployment processes).
*   **Work Processes:** Professional teams work with methodologies such as Kanban, Scrum, or SAFe, which include daily standups, sprint reviews, retrospectives, and mutual code reviews via Pull Requests.
*   **Shared Standards:** The team defines and adheres to coding standards, a unified project structure, and automated testing and deployment processes (CI/CD).

#### 1.3 Software Development Life Cycle
Every professional software project undergoes a defined life cycle (SDLC):
1.  **Requirements Definition:** Writing the Product Requirements Document (PRD).
2.  **Design and Architecture:** System design, writing design documents (PLAN), and setting the roadmap (TODO).

### 2. Mandatory Project Structure and Documentation
Every professional software project **must** include a structure of folders and minimal documentation. Without these documents, the project will not be considered to meet minimum requirements.

#### 2.1 README.md file in the project root
Every project must include a `README.md` file in the root directory. This file serves as a full user manual and must include:
*   **Installation Instructions:** System requirements, step-by-step installation, environment variable configuration, and troubleshooting common issues.
*   **Usage Instructions:** Running the system in different modes, flags, options, CLI/GUI workflows.
*   **Examples and Demonstrations:** Code snippets, screenshots, and common use-case scenarios.
*   **Configuration Guide:** Explanation of configuration files, parameters, and their effects.
*   **Contribution Guidelines:** Code standards and style.
*   **License & Credits:** Usage license and attribution for third-party libraries.

#### 2.2 `docs/` folder — Mandatory documentation
Every project must include a `docs/` folder in the project root, containing at least the following documents:

**1. Product Requirements Document (`docs/PRD.md`)**
The PRD is the central document defining the project's goal and requirements. It includes:
*   Project overview and context, description of the user problem, market analysis, and target audience identification.
*   Measurable goals, KPIs, and acceptance criteria.
*   Functional and non-functional requirements, and user stories.
*   Assumptions, dependencies, and out-of-scope items.
*   Timeline and expected milestones.

**2. Design Document (`docs/PLAN.md`)**
The design document describes the architecture and technical design of the project, including:
*   C4 Model (Context, Container, Component, Code) diagrams.
*   UML diagrams for complex processes and deployment diagrams.
*   Architectural Decision Records (ADRs) with rationale, trade-offs, and alternatives.
*   API documentation, data schemas, and contracts.

**3. Tasks Document (`docs/TODO.md`)**

### 3. Code Documentation and Structure
Proper code documentation and project structure are essential foundations for professional software development. Good documentation allows other developers to quickly understand the code, use it correctly, and contribute efficiently.

#### 3.1 Modular Project Structure
A proper project structure is key to efficient maintenance and future code development. Organizational principles include logical division of the project into directories by function, such as source code, tests, documentation, data, configurations, and resources. The organization can be feature-based or layered architecture, with a clear separation between code, data, results, and documentation.

#### 3.2 File Size Rule — Maximum 150 Lines
**No code file shall exceed 150 lines of code** (empty lines and comment lines are not counted). When a file exceeds this limit, it must be split into multiple files — never compress code to meet the limit.

**Splitting Strategies:**
*   **Extract helper functions** — independent functions to a separate file.
*   **Extract mixins** — when a class has too many responsibilities.
*   **50/50 Split** — when there are two distinct logics (e.g., read/write).
*   **Extract constants** — constants to a `constants.py` file.
*   **Extract models** — definitions to a separate file.

#### 3.3 Code Quality and Comments
Code quality is measured not only by its functionality but also by its readability and maintainability. Code Comment Standards [6], [7], [8] require that comments explain the "why," not just the "what." Every function and module must include detailed Docstrings. Comments should explain complex design decisions, document assumptions and prerequisites, and be updated alongside any code changes.

Principles of high-quality code include using descriptive and accurate names for variables and functions, writing short, focused functions that adhere to the Single Responsibility Principle, avoiding code duplication according to the DRY (Don't Repeat Yourself) principle, and maintaining consistent coding style throughout the project.

### 4. SDK Architecture and Object-Oriented Design

#### 4.1 SDK-Based Architecture
Every function containing business logic must be accessible through the SDK layer. The SDK serves as the sole entry point for all consumers: GUIs, CLIs, third-party integrations, and future services.

**Architectural Diagram:**
```text
External Consumers (GUI / CLI / REST / Third Party)
|
v
+-----------+
|    SDK    | <-- Single entry point for ALL logic
+-----------+
|
v
+-----------+
|  Domain   | <-- Services, models, orchestrators
| Services  |
+-----------+
|
v
+-----------+
| Infrastructure | <-- DB, file I/O, external APIs
+-----------+
```

**Architectural Requirements:**
*   Every business function must be exposed through the SDK.
*   There is no business logic in the GUI or CLI layers — these layers delegate to the SDK.
*   External consumers can only import the SDK and execute all operations without direct access to internal modules.

#### 4.2 Object-Oriented Design (OOP) — No Code Duplication
Design code using the OOP approach. Do not duplicate code. When the same logic appears in two or more files, extract it into a shared module, base class, or mixin.

*   **Same function body in two or more files** — Extract to a shared module.
*   **Same `try/except` pattern in three or more files** — Create a wrapper function.
*   **Same method in three or more classes** — Create a base class or mixin.
*   **Duplicated logic with slight variations** — Use the Template Method design pattern.

### 5. API Gatekeeper and Rate Limiting

#### 5.1 Centralized API Gatekeeper
All external API calls must pass through a centralized gatekeeper. The gatekeeper handles rate limiting, queues, retries, and monitoring.

**Gatekeeper Requirements:**
*   **No direct API calls:** Services must bypass direct calls and use the gatekeeper.
*   **Rate limiting:** Enforced before every call.
*   **Queueing:** Overflow is queued, not rejected.
*   **Monitoring:** All API calls are logged for monitoring.

**API Gatekeeper Interface (Code Block):**
```python
class ApiGatekeeper:
    """Centralized API call manager."""

    def __init__(self, config: RateLimitConfig):
        """Initialize with rate limit config."""
        ...

    def execute(self, api_call, *args, **kwargs):
        """Execute API call through gatekeeper.
        - Check rate limits before execution
        - Queue if limit reached
        - Retry on transient failures
        - Log all calls
        """
        ...

    def get_queue_status(self) -> QueueStatus:
        """Return queue depth and stats."""
        ...
```

#### 5.2 Rate Limit Configuration
All rate limits must be read from a configuration file, never hardcoded in the code.

**Rate Limit Configuration (Code Block):**

### 5.3 Queue Management for Overflow
When rate limits are reached, the gatekeeper must move requests to a queue instead of rejecting them or crashing:

*   **FIFO queue** for pending requests.
*   **Maximum queue depth** defined in the configuration.
*   **Backpressure alert** when the queue is full.
*   **Drain mechanism** that processes requests when the rate limit resets.

### 6. Development-Testing and Quality Assurance

#### 6.1 TDD Process — Red, Green, Refactor
All development must follow the Test-Driven Development (RED — GREEN — REFACTOR) cycle.

*   Every module must include a corresponding test file.
*   Every public function/method must include at least one test.
*   Tests must cover both the standard path and error cases.
*   Tests are written **before** or in parallel to implementation, not as an afterthought.

**Test Structure:**
```text
tests/
unit/
    test_<module>/        # Mirror src/ structure
    test_<file>.py
integration/
    test_<feature>.py
conftest.py               # Shared fixtures
```

**Testing Rules:**
1. Every new module must have a corresponding test file.
2. Every public function must have at least one test.
3. Tests must cover both the standard path and error cases.
4. Use `conftest.py` for shared test fixtures.
5. Mock external dependencies (databases, files, API).
6. Test files must not exceed 150 lines in total.
7. No tests should depend on external services.

#### 6.2 Test Coverage — Minimum 85%
Global test coverage must be 85% or higher. The test suite must fail if coverage drops below this threshold.

**Configuration in `pyproject.toml`:**

**Coverage Configuration**
```toml
[tool.coverage.run]
source = ["src"]
omit = ["src/main.py", "*/tests/*", "src/**/gui/*"]
```

### 7. Code Inspection, Configuration Management, and Information Security

#### 7.1 Linter Compliance — Zero Ruff Errors
**Zero allowed errors.** All code must pass `ruff check` without any errors.
Configuration in `pyproject.toml`:

**Ruff Configuration**
```toml
[tool.ruff]
line-length = 100
target-version = "py310"

[tool.ruff.lint]
select = ["E","F","W","I","N","UP","B","C4","SIM"]
ignore = ["E501"]
```

**Active Categories:**
*   **E** — PEP 8 errors (indentation, whitespace, style)
*   **F** — Pyflakes (undefined names, unused imports)
*   **W** — PEP 8 warnings
*   **I** — isort (import sorting)
*   **N** — pep8-naming (naming conventions)
*   **UP** — pyupgrade (Python 3.10+ modernization)
*   **B** — flake8-bugbear (common bugs)
*   **C4** — Comprehensions usage
*   **SIM** — Simplification

#### 7.2 Prohibition of Hardcoded Values in Code
All configurable values must reach the code via configuration files, not the source code.

**Table 1: Hardcoded vs. Configuration Values**

| Category | Incorrect Usage | Correct Usage |
| :--- | :--- | :--- |
| API Addresses | "https://api.example.com" | cfg.get("api_url") |
| Rate Limits | rate_limit = 10 | cfg.get("rate_limit", 10) |
| Timeouts | timeout=60 | cfg.get("timeout", 60) |
| Secrets | api_key = "abc123" | os.environ.get("API_KEY") |

Allowed values in code: Physical constants, mathematical constants, default values for parameters.

### 7.4 Information Security and Secrets Management
**No secret data in the project. When pushing to GitHub, it is mandatory to create an `.env-example` file with dummy values.**

- It is forbidden to store API keys, passwords, or tokens in the source code [9], [10], [11].
- Use only environment variables: `os.environ.get("API_KEY")`.
- Must include in `.gitignore`: `credentials.json`, `*.key`, `*.pem`, `.env`.
- Must have an `.env-example` file with dummy values.
- In production environments — use secret management tools.
- Rotation of keys, monitoring of usage, and restriction of permissions to the minimum.

### 8. Version Control and `uv` Package Manager

#### 8.1 Global Version Control
The code and configuration files must include explicit version tracking. The version starts at 1.00 and increases with significant changes.

**Table 2: Required Version Locations**

| Item | Location | Initial Value |
| :--- | :--- | :--- |
| Code Version | `src/<pkg>/shared/version.py` | 1.00 |
| Configuration Version | "version" key in JSON | 1.00 |
| Rate Limit Version | "rate_limits.version" | 1.00 |

The application must perform version validation of the configuration during runtime.

#### 8.2 Recommended Workflows with Git
Recommended workflows include maintaining a clear commit history with meaningful messages, using separate branches for new feature development, performing code reviews via Pull Requests, and using tagging to mark major versions.

#### 8.3 Prompt Engineering Log
The development process with AI must include a Prompt Engineering Log. This includes a list of all significant prompts used to build the project, a description of the context and goal, examples of the outputs received, improvements made, and recommended practices derived from the experience.

#### 8.4 `uv` Package Manager — Mandatory
**All projects must use `uv` as the package manager and task runner. It is forbidden to use `pip`, `virtualenv`, or `venv`, or `python -m pip install` directly.**

**Table 3: Required `uv` Commands**

| Task | Correct (uv) | Forbidden |

### 15. Parallel Processing and Performance
The use of multiprocessing and multithreading is essential for the performance of modern software.

#### 15.1 Difference between Multiprocessing and Multithreading
*   **Multiprocessing:** Suitable for CPU-bound operations: mathematical calculations, image processing, model training. Each process operates in a separate memory space and utilizes a different CPU core.
*   **Multithreading:** Suitable for I/O-bound operations: network requests, database access, file reading/writing. Threads allow the execution of other operations while waiting.

#### 15.2 Thread Safety
Thread safety is critical: protecting shared variables using locks, using `queue.Queue` for data transfer, avoiding deadlocks, and using context managers.

#### 15.3 Checklist: Parallel Processing
1. **Operation Identification:** Identifying I/O-bound/CPU-bound operations, choosing the right tool, evaluating benefit.
2. **Implementation:** Number of processes/threads, dynamic sharing of data, correct synchronization.
3. **Resource Management:** Proper closing, handling exceptions, preventing memory leaks.
4. **Safety:** Protecting shared variables, preventing race conditions and deadlocks.

### 16. Modular Design and Building Blocks
Design based on building blocks (building blocks design) is a modular approach where every component is an independent unit with a well-defined interface.

#### 16.1 Building Block Structure
Every building block is defined by:
*   **Input Data:** Data types, valid range, external dependencies, and edge-case validation.
*   **Output Data:** Data types, format, edge-case behaviors.
*   **Setup Data:** Parameters with default values, configuration, initialization.

#### 16.2 Design Principles
*   **Single Responsibility:** Every building block is responsible for one task.
*   **Separation of Concerns:** Every building block deals with one aspect.
*   **Reusability:** Building blocks are independent and not tied to specific code.
*   **Testability:** Every building block can be tested via dependency injection.

#### 16.3 Building Block Example
